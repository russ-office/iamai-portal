# Spec: Пульс пользователя v1 (для Opus Chat)

## Задача для Opus
Собрать из этой спеки **дизайн-бриф + промпт для Claude Design/Fable**: построить interactive-ready HTML Пульс (v1 read-only) пользователя. Мудборд Ruslan даёт в Chat — заложи плейсхолдер «дизайн-направление + мудборд из Chat»; оба Google-Slides выше.

## Продуктовый контур
- **Пульс = персональный дашборд участника Lab.** Архитектура А (M-040): Airtable → `snapshot.json` → HTML+JS. **v1 read-only**; write-proxy (интерактив) = fast-follow.
- **Персона-скоуп** = `pulse_token` (capability-URL, заполнены на Users/Groups/Organizations, 91/9/12). Снапшот фильтрует по токену. Тот же Пульс на 3 уровнях (человек/группа/орг).
- Backend = Airtable `appi7h7PZhQ5riAIu`. Эталон формы данных = `mock_snapshot_pulse_person.json`.

## Экраны (из мокапа)
1. **Обложка:** «Привет, {USER}», {date}, «спринт №{sprint_no}», световой индикатор статуса, статичная иллюстрация.
2. **Бэклог (список):** карточки (заголовок · «N часов/мес» · описание).
3. **Деталь задачи:** календарь · текущий спринт · стадии треда · список задач (To-Do, плашки) · карточка-деталь с полями.
- **Навигация:** Бэклог · Спринт · Библиотека · Marketplace · Leaderboard · История · Календарь.

## Пайплайн лабы (канон Ruslan — семантическое ядро)
Тред Artifacts (T0–T7), с ветвлением:
1. **Бэклог** — список проблем, всё «из головы» (структурированное описание). → T1.
2. **Задача** — **Триада** (структурно): какую проблему решаем · что на ВЫХОДЕ (какая инфо, в каком виде) · что на ВХОДЕ (какая инфо нужна, в каком виде, чтобы прийти к выходу). = слияние 1_problem+2_output_spec+3_input_spec (ратифиц. M-025). → T2.
3. **Исследование** — deep research по Триаде: ищем готовые решения. **Нашли** → в работу (Тесты→ТЗ). **Не нашли/не подходит** → делаем прототип сами. → T3.
4. **Прототип** — JTBD, путь от Входа к Выходу (трансформация информации). Цель — понять, какой путь работает. **Одна задача → много прототипов/гипотез.** Не работает → опыт логируем, понимаем почему. Работает → ТЗ. → T5 (+T4 гипотезы, связь `parent_artifact`/`source_idea`).
5. **Тесты** — проверяем: прототип работает/не работает. → T6.
6. **ТЗ** — структурированное ТЗ. → T7.

## Подтверждено стартовыми презентациями клиента (контекст)
Две колоды первых встреч (Google Slides ниже) канонизируют пайплайн и дают детали:
- **Спринт = 2 недели** («Циклы инноваций»). → `Cycles`.
- **Провал прототипа = ОПЫТ** («знаем, как не работает») — логируется, тоже ценность/активность (`status=not_working`); «работает» → цена/сроки/ROI/ТЗ.
- **Фазы дорожной карты:** phase_1 = внутри департамента (2–3 мес) · phase_2 = между департаментами, «автоматизация на шине» (3–4 мес). → `Groups.phase`.
- **Приоритизация:** цена проблемы vs стоимость решения · быстрые победы vs большие. → `priority_quadrant`/`impact_share`.
- **Demo Day:** «Наше решение ДО/ПОСЛЕ» = before/after часы (T2/T6).
- **Две формы:** Форма 1 = Бэклог · Форма 2 = Задача/Решение (= Fillout `xnyBgL499Qus`/`8RtZLWzrveus`).
- Базовый подход = **JTBD, ВХОД→ВЫХОД** (= Триада). Метод: «думать как инженер», «перемешивание контекста».

## Анатомия карточек (мокап → база)
- **Карточка бэклога:** `title` · «N часов/мес» = `total_hours` (потенциал, `metric_phase=before`) · `content`.
- **Карточка-деталь задачи:** `{ЗАДАЧА ID}`=`artifact_key`/`thread_key` · Проблема/Что на ВЫХОДЕ/Что на ВХОДЕ (Триада) · Время на задачу ДО=`time_per_unit_min` · Кол-во задач в мес=`freq_per_period` · Общие трудозатраты=`total_hours` · Решение(T5)/Тесты(T6)/Техзадание(T7) · «редактировать»=`url_*`.
- **Экономия часов** считается по ТРЕДУ (`thread_key`): before(T2) − after(T6). Дашборд агрегирует по `thread_key`.

## Световой индикатор статуса (v1 — «объективные данные → эмоция»)
Простая 3-цветная плашка, детерминированно из снапшота, на человека. **Пороги — ЗАЛОЧЕНЫ (дефолты Ruslan). ЦВЕТА подбирает Claude Design из мудборда/палитры** (в брифе: маппинг 3 состояний → 3 цвета).
- `days_idle` = сегодня − последняя активность (max: `Artifacts.submitted_at` автора, `LabTasks.done_at` исполнителя, `Sessions.date`); `done_7d` = число завершённых артефактов/задач за 7 дней.
- **Хорошо:** `days_idle ≤ 1` И `done_7d ≥ 2` · **Так себе:** `days_idle 2–3` ИЛИ (свежо, но `done_7d < 2`) · **Плохо:** `days_idle ≥ 4`.
- Залочено: рабочие дни (не календарные); overdue LabTasks — опц. триггер «плохо»; та же сетка на группу/орг (агрегация).
- Данные: `LabTasks.done_at` ДОБАВЛЕН (date, `fldIP9FkxLMQpoQq3`). Заполнение — UI-шаг #618 (конвертнуть в lastModifiedTime по status = нативно, без автоматизации; API этот тип не создаёт).
